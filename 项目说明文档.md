# 大模型性能对比评测系统 — 项目说明文档

## 一、项目概述

本项目是一个基于 **Jakarta EE + MySQL** 架构的 Web 应用，名称为"大模型性能对比评测系统"（LLM Benchmark System）。系统从 MySQL 数据库中读取主流 AI 大模型厂商、模型基础信息和多维性能指标数据，通过 Servlet 处理后，在 JSP 页面中以表格形式展示给用户。用户可通过首页导航卡片分别浏览"厂商列表"、"模型库"、"性能指标"三张数据表，并可在"综合表"页面执行条件筛选、多维度排序和多模型横向对比分析。

### 3.1 数据库名称

`llm_benchmark` — 大模型评测数据库

```
└───────────────┘       └───────────────┘       └──────────────────┘
```

### 3.3 表结构详解

#### （1）creators — 大模型厂商表

```sql
CREATE TABLE `creators` (
  `creator_id`   varchar(10)  NOT NULL COMMENT '厂商ID（如 oai, ant, goog）',
  `creator_name` varchar(50)  NOT NULL COMMENT '厂商全称',
  `description`  text         NULL     COMMENT '厂商简介',
  PRIMARY KEY (`creator_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='大模型厂商/组织表';
```

**字段说明：**
- `creator_id`：主键，极简短 ID，如 `oai`（OpenAI）、`ant`（Anthropic）、`deep`（DeepSeek）
- `creator_name`：厂商完整名称，如 `OpenAI`、`DeepSeek`
- `description`：厂商技术背景与行业地位描述

#### （2）models — 大模型基础信息表

```sql
CREATE TABLE `models` (
  `model_id`            varchar(10)   NOT NULL COMMENT '模型ID（如 gpt55xh, cl47mx）',
  `model_name`          varchar(100)  NOT NULL COMMENT '模型完整名称',
  `creator_id`          varchar(10)   NOT NULL COMMENT '所属厂商ID',
  `context_window`      int           NULL     COMMENT '上下文长度（token）',
  `is_open_source`      tinyint(1)    NULL DEFAULT 0 COMMENT '是否开源',
  `release_date`        date          NULL     COMMENT '发布日期',
  `field_expertise`     varchar(200)  NULL     COMMENT '擅长领域',
  `version_upgrade_note` text         NULL     COMMENT '版本升级说明',
  PRIMARY KEY (`model_id`),
  INDEX `idx_models_creator`(`creator_id`),
  INDEX `idx_models_release_date`(`release_date`),
  INDEX `idx_models_open_source`(`is_open_source`),
  CONSTRAINT `models_ibfk_1` FOREIGN KEY (`creator_id`)
    REFERENCES `creators` (`creator_id`) ON DELETE RESTRICT ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='大模型基础信息表';
```

**关键设计：**
- `context_window` 使用 `INT` 可空类型，因为部分模型未公布上下文长度
- `is_open_source` 使用 `TINYINT(1)` 可空，`NULL` 表示未知
- 外键 `ON DELETE RESTRICT`：若厂商下存在模型，禁止删除该厂商
- 三个索引分别覆盖：所属厂商查询、发布日期排序、开源状态筛选

#### （3）model_metrics — 模型数值指标表（宽表）

```sql
CREATE TABLE `model_metrics` (
  `model_id`            varchar(10)   NOT NULL COMMENT '模型ID',
  `artif_intel_idx`     decimal(5,2)  NULL     COMMENT '智力指数',
  `artif_omni_idx`      decimal(5,2)  NULL     COMMENT '全能指数',
  `terminal_bench_hard` decimal(5,2)  NULL     COMMENT 'Terminal-Bench Hard',
  `aa_omni_accuracy`    decimal(5,2)  NULL     COMMENT 'AA-Omni 准确率',
  `blended_price`       decimal(8,4)  NULL     COMMENT '综合价格($/1M Tokens)',
  `median_tokens_s`     decimal(8,2)  NULL     COMMENT '中位数吞吐量(tok/s)',
  `latency_first_chunk` decimal(8,2)  NULL     COMMENT '首字延迟(s)',
  `total_response_time` decimal(8,2)  NULL     COMMENT '总响应时间(s)',
  PRIMARY KEY (`model_id`),
  INDEX `idx_metrics_intel`(`artif_intel_idx`),
  INDEX `idx_metrics_price`(`blended_price`),
  CONSTRAINT `model_metrics_ibfk_1` FOREIGN KEY (`model_id`)
    REFERENCES `models` (`model_id`) ON DELETE CASCADE ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='模型数值指标表';
```

**关键设计：**
- **宽表设计**：一个模型对应一行，所有指标字段平铺，避免 EAV 模式的行转列开销
- 所有数值字段使用 `DECIMAL` 精确类型：`DECIMAL(5,2)` 表示 5 位有效数字、2 位小数
- `blended_price` 使用 `DECIMAL(8,4)` 保留 4 位小数，满足极小价格精度需求
- 外键 `ON DELETE CASCADE`：删除模型时级联删除其指标数据
- 在 `artif_intel_idx` 和 `blended_price` 上建立索引，优化排序性能

### 4.1 通用全表查询（BaseDAO.findAll）

**所在文件：** `BaseDAO.java`

**SQL：**
```sql
SELECT * FROM {table_name}
```

**Java 代码：**
```java
public List<T> findAll() {
    String sql = "SELECT * FROM " + getTableName();
    try (Connection conn = DBUtil.getConnection();
         Statement stmt = conn.createStatement();
         ResultSet rs = stmt.executeQuery(sql)) {
        while (rs.next()) {
            list.add(mapRow(rs));
        }
    }
}
```

**参数说明：**
- `getTableName()` 是抽象方法，由子类返回具体表名
- 使用 `Statement` 而非 `PreparedStatement`，因为不涉及用户输入，无 SQL 注入风险

**执行场景：**
- 访问 `/list?type=creators` → `CreatorDAO.findAll()` → `SELECT * FROM creators`
- 访问 `/list?type=models` → `ModelDAO.findAll()` → `SELECT * FROM models`
- 访问 `/list?type=metrics` → `ModelMetricDAO.findAll()` → `SELECT * FROM model_metrics`

**实际执行示例：**
```sql
SELECT * FROM creators;      -- 返回 19 条厂商记录
SELECT * FROM models;        -- 返回 64 条模型记录
SELECT * FROM model_metrics; -- 返回 62 条指标记录
```

### 4.2 综合表查询（三表 JOIN）

#### 4.2.1 无条件查询（初始加载）

**所在文件：** `ModelDAO.java` — `findAllWithMetrics()`

**SQL：**
```sql
SELECT m.*, c.creator_name,
       mt.artif_intel_idx, mt.artif_omni_idx,
       mt.terminal_bench_hard, mt.aa_omni_accuracy,
       mt.blended_price, mt.median_tokens_s,
       mt.latency_first_chunk, mt.total_response_time
FROM models m
LEFT JOIN creators c ON m.creator_id = c.creator_id
LEFT JOIN model_metrics mt ON m.model_id = mt.model_id
ORDER BY c.creator_name, m.model_name
```

**查询说明：**
- `m.*`：models 表的所有字段（model_id, model_name, creator_id, context_window 等）
- `c.creator_name`：从 creators 表获取厂商名称用于展示
- `mt.*`：model_metrics 表的 8 个指标字段
- **两次 LEFT JOIN**：
  - 第一次 LEFT JOIN 关联 models → creators：获取厂商名称
  - 第二次 LEFT JOIN 关联 models → model_metrics：获取性能指标
  - 使用 LEFT JOIN 而非 INNER JOIN，确保即使模型缺少指标数据也能显示
- **排序**：按厂商名称升序，再按模型名称升序，保证同一厂商的模型排列在一起

**执行场景：** 用户访问 `/list?type=fullview` 时，`ListServlet` 调用此方法获取初始数据。

#### 4.2.2 条件+排序查询（AJAX 搜索）

**所在文件：** `ModelDAO.java` — `findAllWithMetricsByConditions(Map, orderBy)`

**基础 SQL 骨架：**
```sql
SELECT m.*, c.creator_name,
       mt.artif_intel_idx, mt.artif_omni_idx,
       mt.terminal_bench_hard, mt.aa_omni_accuracy,
       mt.blended_price, mt.median_tokens_s,
       mt.latency_first_chunk, mt.total_response_time
FROM models m
LEFT JOIN creators c ON m.creator_id = c.creator_id
LEFT JOIN model_metrics mt ON m.model_id = mt.model_id
WHERE 1=1
```

`WHERE 1=1` 是动态 SQL 构建的经典技巧——始终为真，后续条件直接用 `AND ...` 拼接，无需判断是否为第一个条件。

##### 条件拼接详解：

**（1）厂商筛选：动态 IN 子句**
```sql
AND m.creator_id IN (?, ?, ?)
```
- **参数**：`List<String> creatorIds`，由前端多选下拉框提供
- **动态生成**：遍历 List，每项添加一个 `?`，中间用逗号分隔
- **示例**：筛选 OpenAI 和 Anthropic → `AND m.creator_id IN ('oai', 'ant')`

**（2）开源状态筛选**
```sql
AND m.is_open_source = 1   -- 仅开源
AND m.is_open_source = 0   -- 仅闭源
```
- `"open"` 映射为 `= 1`，`"closed"` 映射为 `= 0`
- 直接拼接常量，不使用占位符，因为值来自预定义的枚举而非用户输入

**（3）范围条件：通用 addRangeCondition 方法**
```java
private void addRangeCondition(StringBuilder sql, List<Object> params,
                               String column, Object min, Object max) {
    if (min != null && !min.toString().trim().isEmpty()) {
        sql.append(" AND ").append(column).append(" >= ?");
        params.add(parseNumber(min));
    }
    if (max != null && !max.toString().trim().isEmpty()) {
        sql.append(" AND ").append(column).append(" <= ?");
        params.add(parseNumber(max));
    }
}
```

**适用范围字段（6 组）：**

| 前端 ID | 后端参数名 | SQL 列 | 说明 |
|---------|-----------|--------|------|
| `qCtxMin` / `qCtxMax` | `contextWindowMin/Max` | `m.context_window` | 上下文窗口范围 |
| `qIntelMin` / `qIntelMax` | `artifIntelIdxMin/Max` | `mt.artif_intel_idx` | 智力指数范围 |
| `qPriceMin` / `qPriceMax` | `blendedPriceMin/Max` | `mt.blended_price` | 价格范围 |
| `qTpsMin` / `qTpsMax` | `medianTokensSMin/Max` | `mt.median_tokens_s` | 吞吐量范围 |
| `qDateBegin` / `qDateEnd` | `releaseDateBegin/End` | `m.release_date` | 发布日期范围 |

**生成示例**（筛选智力指数 40~60 且价格 ≤ 2.0）：
```sql
AND mt.artif_intel_idx >= ?    -- 参数：40
AND mt.artif_intel_idx <= ?    -- 参数：60
AND mt.blended_price <= ?      -- 参数：2.0
```

`parseNumber()` 辅助方法自动识别整数、小数、日期字符串，避免类型转换异常：
```java
private Object parseNumber(Object val) {
    if (val instanceof Number) return val;
    String s = val.toString().trim();
    if (s.contains(".")) return Double.parseDouble(s);
    return Integer.parseInt(s);
}
```

**（4）擅长领域模糊查询**
```sql
AND m.field_expertise LIKE ?
```
- **参数**：`%` + 用户输入 + `%`，实现任意位置匹配
- **示例**：搜索"代码" → `AND m.field_expertise LIKE '%代码%'`

**（5）排序拼接**
```sql
ORDER BY mt.artif_intel_idx DESC    -- 智力指数降序
ORDER BY m.release_date ASC         -- 发布日期升序
```
- 若未指定排序，则默认 `ORDER BY c.creator_name, m.model_name`
- 排序字段直接拼接（非用户输入），由前端下拉框的固定选项提供，无注入风险

**完整 SQL 示例**（筛选 OpenAI 厂商、智力指数 ≥ 50、按价格升序）：
```sql
SELECT m.*, c.creator_name, mt.artif_intel_idx, mt.artif_omni_idx,
       mt.terminal_bench_hard, mt.aa_omni_accuracy,
       mt.blended_price, mt.median_tokens_s,
       mt.latency_first_chunk, mt.total_response_time
FROM models m
LEFT JOIN creators c ON m.creator_id = c.creator_id
LEFT JOIN model_metrics mt ON m.model_id = mt.model_id
WHERE 1=1
  AND m.creator_id IN ('oai')
  AND mt.artif_intel_idx >= 50
ORDER BY mt.blended_price ASC
```

**执行场景：** 用户在综合表执行条件搜索或切换排序时，前端通过 AJAX 调用 `ManageServlet?action=searchFullview`，后端组装动态 SQL 并返回 JSON。

### 4.3 对比查询（按 ID 批量查询）

**所在文件：** `ModelDAO.java` — `findModelsByIds(List<String>)`

**SQL：**
```sql
SELECT m.*, c.creator_name,
       mt.artif_intel_idx, mt.artif_omni_idx,
       mt.terminal_bench_hard, mt.aa_omni_accuracy,
       mt.blended_price, mt.median_tokens_s,
       mt.latency_first_chunk, mt.total_response_time
FROM models m
LEFT JOIN creators c ON m.creator_id = c.creator_id
LEFT JOIN model_metrics mt ON m.model_id = mt.model_id
WHERE m.model_id IN (?, ?, ?)
```

**参数说明：**
- `List<String> ids`：前端传入的模型 ID 列表（2~5 个），如 `["gpt55xh", "cl47mx", "dsv4pm"]`
- `IN` 子句动态生成与 ID 数量等量的占位符
- 使用 `PreparedStatement.setString()` 设置每个 ID，防止 SQL 注入

**与 findAllWithMetrics 的区别：**
- 表结构 SELECT 和 JOIN 完全相同
- 仅 WHERE 条件不同：`findModelsByIds` 使用 `IN` 子句限定特定 ID

**执行场景：** 用户在综合表勾选 2~5 个模型并点击"开始对比"后，`CompareServlet` 接收 ID 参数并调用此方法。

**实际执行示例：**
```sql
SELECT ... FROM models m
LEFT JOIN creators c ON m.creator_id = c.creator_id
LEFT JOIN model_metrics mt ON m.model_id = mt.model_id
WHERE m.model_id IN ('gpt55xh', 'cl47mx', 'dsv4pm')
```

### 4.5 厂商 CRUD 操作

#### （1）新增厂商

**SQL：**
```sql
INSERT INTO creators (creator_id, creator_name, description)
VALUES (?, ?, ?)
```

**参数：** `creatorId`（主键）、`creatorName`（必填）、`description`（可空）

**校验逻辑（Java 端）：**
- `creatorId`：1~10 位字母或数字，正则 `[a-zA-Z0-9]{1,10}`
- `creatorName`：不能为空

#### （2）更新厂商

**SQL：**
```sql
UPDATE creators SET creator_name = ?, description = ?
WHERE creator_id = ?
```

**注意：** 主键 `creator_id` 不可修改，作为 WHERE 条件

#### （3）删除厂商

**SQL：**
```sql
DELETE FROM creators WHERE creator_id = ?
```

**前置检查：**
```sql
SELECT COUNT(*) FROM models WHERE creator_id = ?
```
- 若返回值 > 0，禁止删除并提示"请先删除该厂商下的所有模型"
- 对应外键 `ON DELETE RESTRICT` 约束

**SQL：**

## 五、各文件功能详解

### 5.1 工具层

#### 【DBUtil.java】— 数据库连接工具类

**核心功能：**
- 使用**静态代码块**在类加载时从 `db.properties` 读取数据库配置
- 通过 `ClassLoader.getResourceAsStream("db.properties")` 从类路径加载
- `getConnection()` 返回 `java.sql.Connection` 对象，所有 DAO 层通过此方法获取连接
- 采用 **try-with-resources** 自动关闭资源

**配置示例（db.properties）：**
```properties
db.driver=com.mysql.cj.jdbc.Driver
db.url=jdbc:mysql://localhost:3306/llm_benchmark?useSSL=false&serverTimezone=UTC&characterEncoding=UTF-8&allowPublicKeyRetrieval=true
db.username=root
db.password=********
```

**重要提示：** `characterEncoding` 必须使用 `UTF-8`（不能写成 `utf8mb4`），否则中文会乱码。

---

### 5.2 实体层

| 实体类 | 映射表 | 特殊处理 |
|--------|--------|---------|
| `Creator.java` | creators | 无特殊处理，全 String 字段 |
| `Model.java` | models | 使用包装类型 `Integer`/`Boolean` 处理可空字段 |
| `ModelMetric.java` | model_metrics | 使用 `BigDecimal` 精确对应 `DECIMAL` 类型 |
| `ModelCompareVO.java` | 三表 JOIN 视图 | 包含 `creatorName` 和全部 8 个指标字段 |

**包装类型选择原因：**
- 数据库允许 NULL 的字段，不能使用 Java 基本类型（int/boolean）
- 包装类型可以为 null，`rs.getObject()` 判断后安全取值
- 示例：`rs.getObject("context_window") != null ? rs.getInt("context_window") : null`

---

### 5.3 数据访问层（DAO）

#### 【BaseDAO.java】— 通用抽象基类

```java
public abstract class BaseDAO<T> {
    public List<T> findAll() { ... }         // 全表查询
    protected abstract String getTableName(); // 子类提供表名
    protected abstract T mapRow(ResultSet rs); // 子类实现行映射
}
```

**模板方法模式：** `findAll()` 封装了完整的 JDBC 流程（连接→执行→遍历→关闭），子类只需提供表名和映射逻辑。

#### 【CreatorDAO.java】— 厂商数据访问

| 方法 | SQL | 说明 |
|------|-----|------|
| `findAll()` | `SELECT * FROM creators` | 继承自 BaseDAO |
| `save(c)` | `INSERT INTO creators ...` | 新增厂商 |
| `update(c)` | `UPDATE creators SET ... WHERE creator_id=?` | 更新厂商信息 |
| `deleteById(id)` | `DELETE FROM creators WHERE creator_id=?` | 删除厂商 |
| `hasRelatedModels(id)` | `SELECT COUNT(*) FROM models WHERE creator_id=?` | 判断是否有关联模型 |

#### 【ModelDAO.java】— 模型数据访问

| 方法 | SQL 核心 | 说明 |
|------|---------|------|
| `findAll()` | `SELECT * FROM models` | 全表查询 |
| `save(m)` | `INSERT INTO models ...` | 新增模型 |
| `update(m)` | `UPDATE models SET ... WHERE model_id=?` | 更新模型 |
| `deleteById(id)` | `DELETE FROM models WHERE model_id=?` | 删除模型（级联删指标） |
| `findAllWithMetrics()` | 三表 LEFT JOIN，全部数据 | 综合表初始加载 |
| `findAllWithMetricsByConditions()` | 三表 LEFT JOIN + 动态条件 + 排序 | 综合表 AJAX 搜索 |
| `findModelsByIds(ids)` | 三表 LEFT JOIN + `WHERE model_id IN (...)` | 对比页数据查询 |
| `findByConditions()` | 两表 LEFT JOIN + 动态条件 | 高级查询（保留扩展） |

#### 【ModelMetricDAO.java】— 指标数据访问

| 方法 | SQL | 说明 |
|------|-----|------|
| `findAll()` | `SELECT * FROM model_metrics` | 全表查询 |
| `save(m)` | `INSERT INTO model_metrics ...` | 新增指标 |
| `update(m)` | `UPDATE model_metrics SET ... WHERE model_id=?` | 更新指标 |
| `deleteById(id)` | `DELETE FROM model_metrics WHERE model_id=?` | 删除指标 |

### 5.4 控制层（Servlet）

#### 【ListServlet.java】— 页面分发控制器

**URL 映射：** `@WebServlet("/list")`

**工作流程：**
1. 接收 `type` 参数：`creators`、`models`、`metrics`、`fullview`
2. 根据 type 调用对应的 DAO 查询方法
3. 将结果存入 `request.setAttribute("dataList", ...)`
4. 通过 `request.getRequestDispatcher(path).forward(req, resp)` 转发到 JSP

**路径映射对照：**

| type 参数 | DAO 调用 | 视图 JSP |
|-----------|---------|---------|
| `creators` | `creatorDAO.findAll()` | creators.jsp |
| `models` | `modelDAO.findAll()` | models.jsp |
| `metrics` | `metricDAO.findAll()` | metrics.jsp |
| `fullview` | `modelDAO.findAllWithMetrics()` | fullview.jsp |

#### 【ManageServlet.java】— CRUD + AJAX API

**URL 映射：** `@WebServlet("/manage")`

**GET 请求：**
- `?action=getCreators` → 返回厂商 JSON 列表（下拉框用）
- `?action=getModelsWithoutMetrics` → 返回无指标的模型 JSON
- `?action=getAllModels` → 返回全部模型 JSON
- `?action=searchFullview&...` → 综合表条件搜索，返回 JSON

**POST 请求：**
- 12 个 CRUD 操作：`addCreator`、`updateCreator`、`deleteCreator`、`addModel`、`updateModel`、`deleteModel`、`addMetric`、`updateMetric`、`deleteMetric`

**JSON 手动构建：** 项目未引入第三方 JSON 库，所有 JSON 响应通过 `StringBuilder` 手动拼接，使用 `escapeJson()` 转义特殊字符。

#### 【CompareServlet.java】— 对比页控制器

**URL 映射：** `@WebServlet("/compare")`

**工作流程：**
1. 接收 `ids` 参数（逗号分隔的模型 ID）
2. 校验数量：2~5 个，否则重定向到首页
3. 调用 `modelDAO.findModelsByIds(ids)` 查询
4. 计算各指标最优值（用于表格高亮）
5. 传递 `source` 参数（用于返回链接）
6. 转发到 `compare.jsp`

---

### 5.5 视图层（JSP）

#### 【fullview.jsp】— 综合表（核心页面）

**功能：**
- 三表 JOIN 数据表格展示
- 可收缩查询面板（厂商下拉框、开源状态、6 组范围筛选、排序）
- 条件搜索 + 重置（AJAX，不刷新页面）
- 模型多选对比（checkbox + `selectedModelIds` Set）
- **已选模型摘要栏**（可折叠，实时显示选中模型标签，支持移除）
- 搜索/筛选/排序/重置操作**不**清除已选模型状态

**JS 核心变量：**
- `selectedModelIds`：`Set`，存储当前勾选的模型 ID
- `modelDataMap`：`Object`，存储 `modelId → {name, creator}` 映射

**关键函数：**

| 函数 | 作用 |
|------|------|
| `onCheckboxChange(cb)` | 勾选/取消时更新 Set，同步全选状态和摘要栏 |
| `searchFullview()` | AJAX 搜索，保留 selectedModelIds，增量更新 modelDataMap |
| `resetFullviewSearch()` | 重置表单后调用 searchFullview()，不清空已选 |
| `renderFullview(items)` | 渲染数据行，恢复已选勾选，增量更新 modelDataMap |
| `toggleAll()` | 全选/取消全选，同步 Set |
| `updateCompareBtn()` | 控制摘要栏内对比按钮的启用/禁用和文案 |
| `startCompare()` | 设置 sessionStorage 标记后跳转对比页 |
| `updateSelectedSummary()` | 根据 Set 渲染摘要栏标签 |
| `removeSelectedModel(id)` | 从摘要栏移除模型（同步取消勾选） |
| `toggleSummaryPanel()` | 展开/收缩摘要栏 |

**对比后清空机制：**
```javascript
// startCompare() 中设置标记
sessionStorage.setItem('compareExecuted', 'true');

// 页面加载时检测标记
if (sessionStorage.getItem('compareExecuted') === 'true') {
    sessionStorage.removeItem('compareExecuted');
    // selectedModelIds 初始即为空 Set，无需显式清除
}
```

#### 【compare.jsp】— 对比分析页

**功能：**
- 选中模型的横向并列表格
- 最优值自动高亮（绿底加粗）
- 差异总结面板：智力最高、性价比最高、吞吐量最高、上下文最大、开源/闭源统计、综合推荐

**返回链接：**
```jsp
<a href="${contextPath}/${source == 'fullview' ? 'list?type=fullview' : 'list?type=models'}">
   ${source == 'fullview' ? '返回综合表' : '返回模型库'}
</a>
```

## 六、请求处理完整流程

### 6.1 综合表初始加载流程

```
用户访问 /list?type=fullview
    ↓
ListServlet.doGet()
    ↓ type=fullview
modelDAO.findAllWithMetrics()
    ↓ 执行 SQL
SELECT m.*, c.creator_name, mt.*
FROM models m
LEFT JOIN creators c ON m.creator_id = c.creator_id
LEFT JOIN model_metrics mt ON m.model_id = mt.model_id
ORDER BY c.creator_name, m.model_name
    ↓ 返回 List<ModelCompareVO>
req.setAttribute("dataList", models)
    ↓
req.getRequestDispatcher("fullview.jsp").forward()
    ↓
fullview.jsp 使用 JSTL <c:forEach> 渲染表格
    + JS 初始化 selectedModelIds（空 Set）和 modelDataMap
```

### 6.2 综合表条件搜索流程（AJAX）

```
用户调整筛选条件 → 点击"搜索"
    ↓
searchFullview() 构建 URLSearchParams
    ↓
fetch('/manage?action=searchFullview&creatorIds=oai&artifIntelIdxMin=50&orderBy=mt.blended_price ASC')
    ↓
ManageServlet.doGet() → action=searchFullview
    ↓ 解析参数到 Map
modelDAO.findAllWithMetricsByConditions(params, orderBy)
    ↓ 动态组装 SQL
SELECT ... WHERE 1=1 AND m.creator_id IN ('oai') AND mt.artif_intel_idx >= 50 ORDER BY mt.blended_price ASC
    ↓ 返回 List<ModelCompareVO>
响应 JSON → renderFullview(data) 更新表格
    ↓
selectedModelIds 保持不变，已选模型被自动恢复勾选
```

### 6.3 模型对比流程

```
用户在综合表勾选 3 个模型 → 点击"开始对比"
    ↓
startCompare()
    → sessionStorage.setItem('compareExecuted', 'true')
    → window.location.href = '/compare?ids=gpt55xh,cl47mx,dsv4pm&source=fullview'
    ↓
CompareServlet.doGet()
    → modelDAO.findModelsByIds(['gpt55xh', 'cl47mx', 'dsv4pm'])
    → 执行 SQL: SELECT ... WHERE m.model_id IN ('gpt55xh', 'cl47mx', 'dsv4pm')
    → 计算各指标最优值
    → 转发 compare.jsp
    ↓
用户查看对比 → 点击"返回综合表"
    ↓
fullview.jsp 重新加载
    → sessionStorage 中 'compareExecuted' 标记被检测并清除
    → selectedModelIds 初始为空 Set
    → 所有复选框未选中
```

## 七、关键技术实践总结

### 7.1 JDBC 最佳实践

1. **try-with-resources**：所有 Connection、Statement、ResultSet 均使用 try-with-resources 自动关闭，避免资源泄漏
2. **PreparedStatement**：所有涉及用户输入的 SQL 均使用参数化查询，防止 SQL 注入
3. **空值安全**：包装类型 + `rs.getObject() != null` 判断，避免 NPE
4. **精确数值**：`BigDecimal` 对应 `DECIMAL` 类型，避免浮点精度问题

### 7.2 动态 SQL 构建

- 使用 `WHERE 1=1` 简化条件拼接逻辑
- `addRangeCondition()` 通用方法处理范围查询
- `IN (...)` 子句根据参数数量动态生成占位符
- 静态排序选项使用直接拼接（非用户输入）

### 7.3 前端状态管理

- `selectedModelIds`（Set）作为勾选状态的唯一数据源
- `modelDataMap` 增量更新，跨搜索保持模型名称映射
- `sessionStorage` 用于跨页面传递"对比已完成"信号
- 所有 AJAX 操作（搜索/排序/重置）均不破坏选择状态

### 7.4 对比按钮合并设计

- 移除顶部独立的对比按钮栏
- 对比按钮仅存在于"已选模型摘要栏"内
- 摘要栏可折叠，未选中模型时自动隐藏
- `updateCompareBtn()` 统一控制按钮状态和文案
